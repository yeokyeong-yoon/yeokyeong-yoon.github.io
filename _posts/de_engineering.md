# 데이터 엔지니어링 핵심 개념 (2년차 SWE 필수 지식)

| 개념 | 정의 | 장점 | 단점 | 적용 예시 |
|------|------|------|------|----------|
| **배치 처리** | 데이터를 일정 기간/크기로 모아 한 번에 처리하는 방식 | • 대용량 처리에 효율적<br>• 리소스 계획 용이<br>• 구현이 상대적으로 간단 | • 실시간성 부족<br>• 지연 시간 발생<br>• 빠른 피드백 어려움 | • 일일 리포트 생성<br>• ETL 파이프라인<br>• 대규모 로그 분석 |
| **스트림 처리** | 데이터를 실시간으로 지속적으로 처리하는 방식 | • 낮은 지연 시간<br>• 실시간 인사이트 제공<br>• 이상 탐지 즉시 가능 | • 구현 복잡도 증가<br>• 리소스 관리 어려움<br>• 장애 복구 복잡 | • 실시간 모니터링<br>• 사기 탐지<br>• 알림 시스템 |
| **ETL** | Extract-Transform-Load: 데이터 추출, 변환, 적재 과정 | • 정형화된 데이터 파이프라인<br>• 데이터 품질 관리 용이<br>• 표준화된 접근법 | • 병목 현상 발생 가능<br>• 확장성 제한될 수 있음<br>• 실시간성 부족 | • 데이터 웨어하우스 구축<br>• 보고서 자동화<br>• 데이터 마이그레이션 |
| **데이터 파티셔닝** | 대용량 데이터를 논리적으로 분할하여 저장/관리 | • 쿼리 성능 향상<br>• 병렬 처리 가능<br>• 관리 용이성 | • 설계 복잡도 증가<br>• 잘못된 설계 시 성능 저하<br>• 데이터 균등 분배 어려움 | • 시간별 데이터 분할<br>• 지역별 데이터 분할<br>• 사용자 ID 기반 분할 |
| **이벤트 소싱** | 상태 변화를 이벤트 시퀀스로 저장/관리하는 패턴 | • 데이터 감사 용이<br>• 상태 재구성 가능<br>• 시스템 디버깅 용이 | • 구현 복잡도 높음<br>• 이벤트 스키마 관리 필요<br>• 쿼리 복잡도 증가 | • 금융 거래 기록<br>• 사용자 액션 추적<br>• 시스템 로깅 |
| **메시지 큐** | 비동기 통신을 위한 메시지 기반 미들웨어 | • 시스템 결합도 감소<br>• 부하 분산 기능<br>• 탄력적 확장성 | • 메시지 순서 보장 어려움<br>• 복잡한 장애 시나리오<br>• 운영 오버헤드 | • 마이크로서비스 통신<br>• 이벤트 기반 처리<br>• 비동기 작업 큐 |
| **데이터 스키마** | 데이터 구조, 형식, 관계 정의 | • 데이터 일관성 보장<br>• 개발 효율성 향상<br>• 쿼리 최적화 용이 | • 변경 관리 어려움<br>• 유연성 제한<br>• 초기 설계 부담 | • 데이터베이스 모델링<br>• API 응답 구조<br>• 데이터 검증 규칙 |
| **캐싱** | 자주 접근하는 데이터 임시 저장 | • 응답 시간 단축<br>• 서버 부하 감소<br>• 비용 효율적 | • 일관성 문제 발생 가능<br>• 캐시 무효화 복잡성<br>• 메모리 관리 필요 | • DB 쿼리 결과 캐싱<br>• API 응답 캐싱<br>• 계산 결과 캐싱 |
| **데이터 복제** | 데이터를 여러 위치에 복사하여 저장 | • 고가용성 보장<br>• 지역적 접근성 향상<br>• 부하 분산 | • 일관성 유지 어려움<br>• 복제 지연 발생<br>• 리소스 사용량 증가 | • 읽기 전용 복제본<br>• 지역 분산 데이터<br>• 재해 복구 시스템 |
| **OLTP vs OLAP** | 트랜잭션 처리 vs 분석 처리 시스템 | • 목적에 최적화된 구조<br>• 성능 향상<br>• 리소스 효율적 사용 | • 시스템 복잡도 증가<br>• 데이터 동기화 필요<br>• 운영 비용 증가 | • OLTP: 주문 처리<br>• OLAP: 판매 분석<br>• 데이터 마트 구축 |
| **데이터 품질 관리** | 데이터의 정확성, 완전성, 일관성 보장 | • 신뢰할 수 있는 분석<br>• 오류 감소<br>• 비즈니스 결정 품질 향상 | • 추가 검증 단계 필요<br>• 처리 시간 증가<br>• 구현 비용 | • 자동화된 데이터 검증<br>• 이상치 탐지<br>• 메타데이터 관리 |
| **분산 시스템** | 여러 노드에 작업을 분산 처리하는 시스템 | • 확장성 향상<br>• 내결함성 증가<br>• 성능 향상 | • 구현 복잡도 높음<br>• 디버깅 어려움<br>• 일관성 관리 복잡 | • 하둡 에코시스템<br>• 클라우드 서비스<br>• 마이크로서비스 |
| **데이터 마스킹** | 민감 데이터 보호를 위한 변환 기법 | • 개인정보 보호<br>• 규제 준수<br>• 보안 위험 감소 | • 작업 복잡도 증가<br>• 성능 영향<br>• 사용성 감소 | • PII 익명화<br>• 테스트 데이터 생성<br>• 개발 환경 데이터 |
| **SQL vs NoSQL** | 관계형 vs 비관계형 데이터베이스 | • 사용 사례별 최적화<br>• 다양한 데이터 모델 지원<br>• 유연한 선택 가능 | • 기술 복잡도 증가<br>• 다중 데이터베이스 관리<br>• 일관된 접근 어려움 | • SQL: 트랜잭션 데이터<br>• NoSQL: 로그, 문서 저장<br>• 하이브리드: 마이크로서비스 |
| **오프셋 관리** | 스트림 처리에서 데이터 위치 추적 | • 정확한 데이터 처리<br>• 장애 복구 용이<br>• 중복 처리 방지 | • 구현 복잡도<br>• 성능 영향<br>• 상태 관리 필요 | • Kafka 소비자 그룹<br>• 체크포인트 기반 처리<br>• 실패 지점부터 재시작 |
| **멱등성** | 동일 작업 여러번 수행해도 결과 동일 | • 신뢰성 향상<br>• 재시도 메커니즘 단순화<br>• 장애 복구 용이 | • 구현 복잡도 증가<br>• 성능 오버헤드<br>• 시스템 설계 제약 | • 결제 처리<br>• 데이터 파이프라인<br>• API 엔드포인트 설계 |

## 핵심 데이터 기술

| 기술 | 주요 용도 | 장점 | 단점 | 사용 시나리오 |
|------|----------|------|------|--------------|
| **Apache Kafka** | 대규모 실시간 데이터 스트리밍 | • 고성능 처리<br>• 내구성과 안정성<br>• 확장성 | • 운영 복잡도<br>• 튜닝 어려움<br>• 클러스터 관리 부담 | • 로그 수집<br>• 이벤트 스트리밍<br>• 시스템 간 통합 |

| **Redis** | 인메모리 데이터 저장소 | • 초고속 응답<br>• 다양한 데이터 구조<br>• 간단한 사용법 | • 메모리 제한<br>• 영속성 이슈<br>• 복잡한 분산 설정 | • 세션 저장<br>• 캐싱<br>• 실시간 순위표 |

| **Spark** | 대규모 데이터 처리 | • 메모리 내 처리<br>• 다양한 연산 지원<br>• 배치/스트림 통합 | • 리소스 요구량 큼<br>• 튜닝 복잡성<br>• 디버깅 어려움 | • 대용량 데이터 분석<br>• 머신러닝 파이프라인<br>• 복잡한 ETL |
| **Elasticsearch** | 검색 및 분석 엔진 | • 빠른 전문 검색<br>• 실시간 분석<br>• 분산 아키텍처 | • 리소스 집약적<br>• 클러스터 관리 복잡<br>• 데이터 일관성 이슈 | • 로그 분석<br>• 검색 기능<br>• 실시간 모니터링 |
| **Docker/K8s** | 컨테이너 관리 | • 환경 일관성<br>• 배포 단순화<br>• 리소스 효율성 | • 학습 곡선<br>• 운영 복잡성<br>• 디버깅 어려움 | • 마이크로서비스<br>• 개발-운영 환경 통일<br>• 확장 가능한 서비스 |
